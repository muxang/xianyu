# 模块 9：出站调节（去 AI 味 + 打字延迟 + 分段）

## 9.1 目标与边界

**做什么**：
- **去 AI 味改写**：把 LLM 的客服腔改成闲鱼卖家口语
- **消息分段**：长回复拆成多条
- **打字延迟**：根据文本长度计算合理延迟
- **最终频控**：单账号/单会话发送频率上限

**不做什么**：
- 不改变语义（只改风格）
- 不做新生成（只润色）
- 不决定"发不发"（自动化分档的事）

**本质**：**拟人化最后一公里**。让 AI 输出看起来像真人发出。

## 9.2 外部契约

**上游**：通过合规检查的 reply_text

**下游**：消息流水线 out_queue（delayed）

**核心 API**：

```python
async def shape_outbound(
    reply_text: str,
    context: StructuredContext,
    automation_level: AutomationLevel,
) -> list[OutboundMessage]
```

返回**多条**消息（分段后），每条带 `send_at`。

## 9.3 去 AI 味改写

### 9.3.1 Prompt

```
[System]
你是闲鱼卖家小助手，任务是把客服式回复改成闲鱼卖家的日常口语。

改写要求：
1. 去掉"您好"、"请问"、"感谢咨询"、"为您服务"等客服腔
2. 多用语气词（哈、呢、哦、～、呀）
3. 短句为主，一句话不超过 20 字
4. 保留所有事实信息（价格、商品、承诺）不变
5. 保留数字、商品名完全一致
6. 不要加 Emoji 过多（全回复最多 1-2 个）

改写前：{original_reply}

改写后（直接输出，不要解释）：
```

### 9.3.2 风格个性化

PersonaConfig 的 `style_tags`：
- `亲切`：多用"亲"、"～"、柔和
- `专业`：偏干练、少语气词
- `幽默`：玩笑、表情
- `简短`：刻意短句

注入 prompt 得到不同风格输出。

### 9.3.3 模型选择

- **Qwen-Flash**：首选够用
- 改写失败（太离谱 / 改动数字）→ fallback 用原文

### 9.3.4 安全检查

改写后要**再过一次风控**：
- 改写可能意外引入承诺词（如"性价比高"→"保证划算"）
- 改写后违规 → 用原文

## 9.4 消息分段

### 9.4.1 规则

```
总长度 < 30 字：不分段
30-60 字：可能分 2 段（看分割点）
60-100 字：分 2-3 段
> 100 字：分 3 段（生成层本应控制 < 80 字）
```

**分割点识别**：
- 句号、问号、感叹号 → 优先
- 逗号 → 次要
- "～"、"哈"、"哦" → 弱分割

**每段长度**：15-30 字

### 9.4.2 目的

1. 模拟真人"说完一句想想再说"
2. 连发多条比一大段像活人
3. 手机端阅读体验好

## 9.5 打字延迟计算

```python
def calculate_delay(text: str, is_first_segment: bool, prev_delay: int) -> int:
    """返回延迟毫秒"""
    # 基础：4-6 字/秒 = 170-250 ms/字
    base_per_char = random.randint(170, 250)
    text_delay = len(text) * base_per_char

    # 思考时间
    thinking = random.randint(2000, 5000) if is_first_segment else random.randint(500, 1500)

    # 段间间隔
    segment_gap = random.randint(1500, 3500) if not is_first_segment else 0

    # 最小 3 秒
    total = max(3000, thinking + text_delay + segment_gap)
    # 最大 30 秒单段
    total = min(30000, total)

    return total
```

**边界**：
- 深夜时段：基础延迟加倍（模拟"刚起来"）
- 买家连发 3 条 → 合并响应，不逐条回

## 9.6 频控

- 单账号：每分钟 ≤ 15 条（含分段）
- 单会话：每分钟 ≤ 6 条
- 突破：后续消息排到下分钟

## 9.7 关键决策

| 决策 | 选择 | 理由 |
|-----|------|-----|
| 改写模型 | Qwen-Flash | 简单任务快便宜 |
| 改写失败 | 原文发送 | 不冒险改错 |
| 分段策略 | 规则 + 句子边界 | 简单可控 |
| 延迟模型 | 规则 + 随机 | 真人本就随机 |
| 深夜调整 | 加倍延迟 | 业务时段差异 |

## 9.8 边界情况

- 改写后字数相差太多（> ±30%）→ 视为失败用原文
- 改写出现原文没有的数字 → 失败
- 改写出现原文没有的承诺词 → 失败
- 消息只有 2 字（"好的"）→ 不改写不分段
- 图片消息不走本模块（直接发）

## 9.9 测试要求

- 改写质量（人工评 20 条）：客服感明显降低
- 数字一致性（改写前后数字相同）
- 分段合理（不切断数字/商品名）
- 延迟分布符合真人打字

## 9.10 实施指引

**实施顺序**：
1. 改写 prompt + 调用
2. 改写后自检（数字、长度、合规）
3. 分段算法
4. 延迟计算
5. 频控（Redis 计数器）
6. 端到端测试

**不要做**：
- ❌ 让改写模型"发挥创意"加内容
- ❌ 所有消息都分段（短消息不需要）
- ❌ 固定延迟（太机械）

- `backend/app/modules/outbound_shaping/`
- `style_rewriter.py`、`segmenter.py`、`delay_calculator.py`、`rate_limiter.py`
- `shaper.py`（主入口）
- `prompts/style_rewrite.md`
